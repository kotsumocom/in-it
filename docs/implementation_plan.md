# 実装計画 - フェーズ 2: コア機能の実装

## 目的 (Goal)

MVP のコア機能である「ユーザー認証」「データ API」「学習マップ UI」を実装します。

## ユーザー確認事項 (User Review Required)

> [!NOTE]
> ユーザー管理には Supabase Auth を利用します。学習進捗を保存するためにはログイン/サインアップが必要になります。

## 提案される変更 (Proposed Changes)

### 1. 認証機能 (Supabase Auth)

- **Backend:** Supabase クライアントは準備済みです。
- **Web (Fresh):**
  - `/login` (ログイン) と `/signup` (新規登録) のページを作成します。
  - 認証状態を管理し、保護されたページへのアクセスを制御するミドルウェア (Middleware) を実装します。
  - ブラウザ側での認証フロー（Supabase JS クライアント利用）には `islands` を使用します。

### 2. API 開発 (Hono & Backend Logic)

- **`backend/`:** 以下のデータ操作ロジックを実装します。
  - `fetchUnits()`: 学習単元の階層構造を取得。
  - `fetchProgress(userId)`: ユーザーの進捗を取得。
  - `updateProgress(userId, unitId, status)`: 進捗を更新（完了など）。
- **`backend/main.ts`:** 上記ロジックを外部（将来のアプリ等）から呼べる API エンドポイントとして公開します。

### 3. "The Map" UI (Fresh)

- **デザイン:** 学習単元をツリー状（またはロードマップ状）に可視化します。
  - 親子関係がわかるようなシンプルな UI コンポーネントを作成します。
- **データ取得:**
  - Fresh のルートローダーから、バックエンドのロジック（`backend/`内の関数）を直接インポートして呼び出します。
  - これにより、モノレポ構成内で型安全かつ高速にデータを取得できます（自分自身の API を HTTP 経由で叩くオーバーヘッドを回避）。

## 検証計画 (Verification Plan)

### 自動テスト

- ログイン後、ダッシュボードへ適切にリダイレクトされるか検証します。
- DB のデータを読み込み、学習マップが正しい階層構造で表示されるかテストします。

### 手動検証

- 新規ユーザーでサインアップを行い、`profiles` テーブルにレコードが作成されるか確認します。
- UI 上で単元を「完了」にし、`user_progress` テーブルが更新されるか確認します。
